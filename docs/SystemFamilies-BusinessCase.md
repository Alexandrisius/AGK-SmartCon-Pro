# Business Case: Системные семейства в FamilyManager

## Статус
Черновик. Предварительный план после мозгового штурма.

---

## Проблема

FamilyManager (FM) на данный момент поддерживает только **загружаемые семейства** (.rfa), созданные в Редакторе семейств Revit. Однако в проектах активно используются **системные семейства** (стены, перекрытия, трубы, воздуховоды, лотки, короба, лестницы, крыши и др.), которые невозможно сохранить как .rfa и загрузить стандартным способом.

Инженерам-проектировщикам необходимо:
- Хранить библиотеку системных типов (типоразмеров) для быстрого переиспользования между проектами.
- Загружать системные типы в проект из каталога FM одной командой.
- Размещать элементы системных семейств (например, нарисовать трубу нужного типа) напрямую из FM.

---

## Решение

Расширить FamilyManager поддержкой системных типов Revit. Системные типы будут храниться в каталоге FM наравне с загружаемыми семействами, но с другой логикой импорта и размещения.

### Принцип работы

1. **Импорт**: пользователь открывает проект-источник (.rvt), выбирает элемент/тип в Revit — FM извлекает его системный тип и сохраняет в каталог.
2. **Хранение**: исходный проект (.rvt) сохраняется в managed storage как "шаблон-источник". В БД FM хранится ссылка на тип внутри этого проекта.
3. **Размещение**: при команде "Загрузить и разместить" FM копирует системный тип из шаблонного .rvt в активный проект и активирует стандартный инструмент Revit для размещения элементов этого типа.

---

## Пользовательские сценарии

### UC-1: Импорт системного типа в каталог

**Актор**: BIM-менеджер / инженер-проектировщик.

**Шаги**:
1. Пользователь открывает в Revit проект-источник, содержащий нужный системный тип (например, "Медная труба").
2. В FM нажимает стрелку рядом с кнопкой "Импорт" — выбивает "Импорт системного семейства".
3. FM переключается в режим пикера: пользователь кликает на элемент с нужным типом в проекте.
4. FM извлекает тип элемента, создаёт запись в каталоге и сохраняет исходный .rvt в managed storage (если он новый).
5. Тип появляется в каталоге FM в категории "Без категории" (пользователь может переместить в любую кастомную категорию).

**Результат**: системный тип доступен в каталоге FM для последующего использования.

---

### UC-2: Размещение системного типа в проекте

**Актор**: Инженер-проектировщик.

**Шаги**:
1. Пользователь находит в каталоге FM нужный системный тип (например, "Базовая стена 200мм").
2. Нажимает "Загрузить и разместить".
3. FM копирует тип из шаблонного .rvt в активный проект (если типа с таким именем ещё нет).
4. Активируется стандартный инструмент Revit для размещения элементов данного типа (например, инструмент "Стена").
5. Пользователь размещает элементы в проекте стандартным способом Revit.

**Результат**: элементы системного типа размещены в проекте.

---

### UC-3: Извлечение атрибутов системного типа

**Актор**: BIM-менеджер.

**Шаги**:
1. Пользователь выбирает системный тип в каталоге FM.
2. Нажимает "Импорт данных" (существующая команда FM).
3. FM открывает шаблонный .rvt, находит тип по UniqueId и извлекает параметры типа в соответствии с настроенным пресетом категории.
4. Атрибуты отображаются в окне "Свойства" FM.

**Результат**: параметры системного типа доступны для просмотра в FM.

---

### UC-4: Обновление системного типа

**Актор**: BIM-менеджер.

**Шаги**:
1. Пользователь вносит изменения в системный тип в проекте-источнике.
2. Повторно запускает "Импорт системного семейства" и выбирает обновлённый элемент.
3. FM обновляет запись в каталоге (ручное обновление, автоматического отслеживания нет).

**Результат**: каталог FM содержит актуальную версию типа.

---

## Ключевые решения

| Аспект | Решение |
|---|---|
| **Источник системных типов** | Файлы .rvt (проекты Revit), сохраняемые в managed storage как шаблоны-источники. |
| **Идентификация типа в шаблоне** | `UniqueId` элемента (стабильный GUID). `ElementId` не используется, т.к. нестабилен между сессиями. |
| **Копирование в проект** | `ElementTransformUtils.CopyElements` из .rvt-шаблона в активный проект. Копирует тип вместе со всеми зависимостями (материалы, сегменты, фитинги). |
| **Конфликт имён** | Если тип с таким именем уже есть в проекте — пропускаем (skip), активируем существующий. |
| **Активация инструмента** | `UIDocument.PostRequestForElementTypePlacement(ElementType)` — стандартный инструмент Revit для размещения элементов данного типа. |
| **Мульти-версионность** | Как у загружаемых семейств — определяем мажорную версию Revit проекта-источника и сохраняем в папку версии (`R25/`, `R21/` и т.д.). |
| **Типоразмеры (MEP)** | Не отображаются. Системный тип показывается как единая сущность (например, "Медная труба"). Пользователь выбирает размер в Property Palette Revit во время размещения. |
| **Атрибуты** | Извлекаются отдельной командой "Импорт данных" (ручной запуск), не при импорте. |
| **Превью** | Не генерируется автоматически. Пользователь может добавить вручную, как у загружаемых семейств. |
| **Категории в FM** | Системные типы загружаются в "Без категории". Пользователь перемещает в кастомные категории вручную. |

---

## Границы решения (Out of Scope)

- Автоматическое отслеживание изменений в шаблонных .rvt.
- Генерация превью/иконок для системных типов.
- Отображение MEP-размеров (сегментов) как под-узлов в дереве FM.
- Предустановка диаметра MEP перед размещением (ограничение API Revit).
- Удаление дублирующихся материалов при копировании между проектами.

---

## Критерии приёмки

1. Пользователь может импортировать системный тип из открытого .rvt-проекта в FM.
2. Импортированный тип отображается в каталоге FM и доступен для поиска.
3. Пользователь может разместить системный тип в активном проекте командой "Загрузить и разместить".
4. Системный тип корректно копируется в проект вместе с зависимостями.
5. Если тип уже существует в проекте — FM активирует существующий, не создавая дубликатов.
6. Атрибуты системного типа извлекаются командой "Импорт данных" и отображаются в окне свойств.
7. Поддерживаются все основные системные семейства: стены, перекрытия, трубы, воздуховоды, лотки, короба, лестницы, крыши.
