# Business Case: Системные семейства в FamilyManager

## Статус
Утверждён. Детальный план после технического исследования и мозгового штурма.

---

## Проблема

FamilyManager (FM) поддерживает только **загружаемые семейства** (.rfa), созданные в Редакторе семейств Revit. Однако в проектах активно используются **системные семейства** (стены, перекрытия, трубы, воздуховоды, лотки, короба, лестницы, крыши и др.), которые невозможно сохранить как .rfa и загрузить стандартным способом.

Инженерам-проектировщикам необходимо:
- Хранить библиотеку системных типов (типоразмеров) для быстрого переиспользования между проектами.
- Загружать системные типы в проект из каталога FM одной командой.
- Размещать элементы системных семейств (например, нарисовать трубу нужного типа) напрямую из FM.

---

## Решение

Расширить FamilyManager поддержкой системных типов Revit. Системные типы хранятся в каталоге FM наравне с загружаемыми семействами по модели: **чистый .rvt файл = семейство, ElementType'ы внутри = типоразмеры**.

### Принцип работы

1. **Импорт**: пользователь выделяет элементы в любом открытом проекте Revit → FM автоматически разбивает выделенное по категориям → показывает диалог с таблицей (категория + имя семейства) → для каждой категории создаёт чистый .rvt проект → копирует туда типы → сохраняет в managed storage.
2. **Хранение**: чистый .rvt файл (200 КБ — 2 МБ) в managed storage по схеме аналогичной .rfa. В БД хранятся ссылки на типоразмеры (UniqueId) внутри этого проекта.
3. **Размещение**: при команде "Загрузить и разместить" FM открывает .rvt из storage через `OpenDocumentFile`, копирует выбранный типоразмер в активный проект через `CopyElements` и активирует стандартный инструмент Revit.

### Поддерживаемые системные семейства

| ElementTypeGroup | API-класс | BuiltInCategory | Особенности |
|---|---|---|---|
| WallType | `WallType` | `OST_Walls` | CompoundStructure, материалы слоёв |
| FloorType | `FloorType` | `OST_Floors` | CompoundStructure, материалы слоёв |
| RoofType | `RoofType` | `OST_Roofs` | CompoundStructure, материалы слоёв |
| PipeType | `PipeType` | `OST_PipeCurves` | RoutingPreferenceManager, Segment, фиттинги |
| DuctType | `DuctType` | `OST_DuctCurves` | RoutingPreferenceManager, Segment, фиттинги |
| FlexPipeType | `FlexPipeType` | `OST_FlexPipeCurves` | Гибкие трубы |
| FlexDuctType | `FlexDuctType` | `OST_FlexDuctCurves` | Гибкие воздуховоды |
| ConduitType | `ConduitType` | `OST_Conduit` | Короба |
| CableTrayType | `CableTrayType` | `OST_CableTray` | Лотки |
| DuctInsulationType | `DuctInsulationType` | `OST_DuctInsulations` | Изоляция воздуховодов |
| PipeInsulationType | `PipeInsulationType` | `OST_PipeInsulations` | Изоляция труб |
| StairsType | `StairsType` | `OST_Stairs` | Лестницы |

## Пользовательские сценарии

### UC-1: Импорт системных семейств в каталог

**Актор**: BIM-менеджер / инженер-проектировщик.

**Предусловие**: В Revit открыт проект с нужными системными типами.

**Шаги**:

1. Пользователь нажимает стрелку рядом с кнопкой "Импорт" в FM — выбирает "Импорт системного семейства".
2. FM переключается в режим пикера: пользователь выделяет один или несколько элементов в проекте.
3. Пользователь подтверждает выбор (Enter / кнопка "Готово").
4. FM извлекает уникальные `ElementType` из выделенных элементов.
5. FM группирует типы по `BuiltInCategory`. Если выделены элементы разных категорий — каждая категория станет отдельным семейством.
6. FM показывает **диалог импорта** — таблица:

   | Категория (readonly) | Имя семейства (редактируемое) | Типов выделено (readonly) |
   |---|---|---|
   | Трубы | _Трубы проект А_ | 5 |
   | Воздуховоды | _Воздуховоды проект А_ | 3 |

7. Пользователь задаёт имена для каждой категории и нажимает "Импорт".
8. **Для каждого семейства** (по категории), в контексте Revit API через `ExternalEvent`:
   a. FM создаёт чистый проект: `app.NewProjectDocument(UnitSystem.Metric)`.
   b. Собирает набор `ElementId` для копирования: типы + зависимости.
      - Для MEP-типов: PipeType/DuctType + Segment (из RoutingPreferenceManager) + PipeScheduleType + Material.
      - Для Host-типов: WallType/FloorType/RoofType (материалы копируются автоматически как зависимости CompoundStructure).
   c. Копирует через `ElementTransformUtils.CopyElements(sourceDoc, elementIds, newDoc, Transform.Identity, options)`.
   d. Сохраняет: `newDoc.SaveAs(tempPath)` → закрывает → перемещает в managed storage.
   e. Вычисляет SHA256 файла.
   f. Создаёт записи в БД: `catalog_items` + `catalog_versions` + `family_files` + `family_types`.
9. Семейства появляются в каталоге FM в категории "Без категории".
10. Пользователь может переместить их в нужную категорию дерева FM.

**Результат**: одно или несколько системных семейств с типоразмерами доступны в каталоге FM.

**Ограничения**:
- Если выделены элементы только загрузаемых семейств (не системных) — FM показывает сообщение "Выделенные элементы не содержат системных типов".
- Максимальное количество выделяемых элементов — не ограничено, но FM извлекает только **уникальные** типы.
- Версия .rvt файла = текущая версия Revit. Файл НЕ обратно совместим (R25 → R24 не работает).

**Контроль дублей при импорте**:
- FM проверяет по SHA256: если файл с таким хешем уже есть в storage → `WasSkippedAsDuplicate`.
- FM проверяет по имени: если семейство с таким именем уже существует → создаёт новую версию (UC-4).

### UC-2: Размещение типоразмера в проекте

**Актор**: Инженер-проектировщик.

**Предусловие**: В Revit открыт проект-цель. В FM есть системное семейство с типоразмерами.

**Шаги**:

1. Пользователь находит в каталоге FM системное семейство (например, "Трубы проект А").
2. Раскрывает его и выбирает нужный типоразмер (например, "Медная труба DN50").
3. Нажимает "Загрузить и разместить".
4. Через `ExternalEvent`, в контексте Revit API:
   a. FM резолвит файл: `IFileResolver.ResolveForLoadAsync(catalogItemId, currentRevitVersion)` → абсолютный путь к .rvt в managed storage.
   b. FM открывает .rvt как фоновый документ: `app.OpenDocumentFile(rvtPath)`.
   c. Находит тип по `UniqueId`: `sourceDoc.GetElement(uniqueId)` → `ElementType`.
   d. **Проверка дубликата**: ищет тип с таким же `Name` в активном проекте через `FilteredElementCollector`.OfClass(typeof(ElementType)).OfCategory(category).
   e. **Если тип уже существует** (совпадение имени):
      - Пропускаем копирование — используем существующий тип.
      - Закрываем sourceDoc.
      - Переходим к шагу 5 с существующим типом.
   f. **Если тип новый**:
      - Собирает полный набор зависимостей (аналогично UC-1 шаг 8b).
      - Копирует через `ElementTransformUtils.CopyElements` с `IDuplicateTypeNamesHandler` = `UseDestinationTypes`.
      - Подавляет предупреждение `BuiltInFailures.CopyPasteFailures.CannotCopyDuplicates` через `IFailuresPreprocessor`.
      - Находит скопированный тип в активном проекте (по имени, т.к. возвращаемые ID не упорядочены).
      - Закрывает sourceDoc.
5. FM активирует стандартный инструмент размещения: `uidoc.PostRequestForElementTypePlacement(elementType)`.
6. Revit показывает стандартный UI инструмента (Pipe Tool, Wall Tool и т.д.).
7. Пользователь размещает элементы стандартным способом Revit.
8. FM записывает использование: `IUsageRepository.RecordUsageAsync("LoadAndPlace")`.

**Результат**: элементы выбранного типоразмера размещены в проекте.

**Технические нюансы**:

- `PostRequestForElementTypePlacement` **нельзя** вызывать внутри активной транзакции — метод создаёт свою.
- Revit автоматически определяет нужный инструмент по категории типа (PipeType → Pipe Tool, WallType → Wall Tool).
- API не уведомляет о завершении размещения. При необходимости отслеживания — `IUpdater` или `DocumentChanged`.
- Для MEP-элементов: PipingSystemType/MechanicalSystemType **не контролируется** FM. Пользователь выбирает систему вручную при размещении через стандартный UI Revit.
- Для MEP-элементов: диаметр/размер **не предустанавливается** FM. Пользователь выбирает размер через стандартный UI Revit.

### UC-3: Извлечение атрибутов типоразмера

**Актор**: BIM-менеджер.

**Предусловие**: В FM настроен пресет атрибутов для категории системного семейства (например, "MEP Трубы" с параметрами: диаметр, материал, соединение).

**Шаги**:

1. Пользователь выбирает типоразмер в каталоге FM.
2. Нажимает "Импорт данных" (существующая команда FM).
3. Через `ExternalEvent`, в контексте Revit API:
   a. FM резолвит .rvt из storage.
   b. Открывает .rvt как фоновый документ: `app.OpenDocumentFile(rvtPath)`.
   c. Находит тип по `UniqueId`.
   d. Читает параметры типа в соответствии с пресетом категории.
   e. Закрывает sourceDoc.
4. Извлечённые значения сохраняются в `extracted_attribute_values` (существующая схема).
5. Атрибуты отображаются в окне "Свойства" FM (вкладка "Атрибуты", read-only).

**Результат**: параметры типоразмера доступны для просмотра в FM.

**Отличие от .rfa**: Для .rfa FM открывает .rfa через `Application.OpenDocumentFile` и читает `FamilyManager.Types`. Для .rvt FM открывает .rvt через `Application.OpenDocumentFile` и читает параметры `ElementType` напрямую. Механизм пресетов и отображения — идентичный.

### UC-4: Дозагрузка типоразмеров (обновление семейства)

**Актор**: BIM-менеджер.

**Предусловие**: Семейство "Трубы проект А" уже существует в FM.

**Шаги**:

1. В открытом проекте появились новые типы (или изменились существующие).
2. Пользователь выделяет элементы (старые + новые) и запускает "Импорт системного семейства".
3. FM показывает диалог импорта. Пользователь вводит имя **существующего** семейства ("Трубы проект А").
4. FM обнаруживает, что семейство с таким именем уже существует в БД.
5. В контексте Revit API:
   a. FM открывает существующий .rvt из managed storage.
   b. Копирует новые типы из текущего проекта в существующий .rvt.
   c. Определяет изменения:
      - Новые типы (нет в существующем .rvt) → добавляются.
      - Изменённые типы (имя совпадает, параметры отличаются) → обновляются (удаляем старый тип, копируем новый).
      - Существующие типы без изменений → пропускаются.
   d. Сохраняет обновлённый .rvt (новая версия файла).
   e. Вычисляет SHA256 новой версии.
   f. Создаёт новую запись `catalog_versions` + обновляет `family_types`.
6. В дереве FM семейство показывает обновлённый список типоразмеров.
7. Старые типоразмеры, не выбранные при повторном импорте, **сохраняются** в семействе.

**Результат**: семейство в FM содержит актуальный набор типоразмеров.

---

### UC-5: Загрузка без размещения (Load Only)

**Актор**: Инженер-проектировщик.

**Шаги**:

1. Пользователь выбирает типоразмер в каталоге FM.
2. Нажимает "Загрузить" (без размещения).
3. FM копирует тип в активный проект (аналогично UC-2 шаги 4a-4f), но **не** вызывает `PostRequestForElementTypePlacement`.
4. Тип доступен в проекте через стандартный Type Selector Revit.

**Результат**: тип загружен в проект и доступен через стандартные инструменты Revit.

## Ключевые решения

### Хранение и идентификация

| Аспект | Решение | Обоснование |
|---|---|---|
| **Модель хранения** | Один чистый .rvt файл = одно семейство в FM (аналог .rfa). | Единая модель для загружаемых и системных семейств. |
| **Создание .rvt** | `app.NewProjectDocument(UnitSystem.Metric)` + `CopyElements` типов + `SaveAs`. | Минимальный файл (~200 КБ), нет мусора. FM сам контролирует создание. |
| **Типоразмеры** | `ElementType` внутри .rvt = типоразмеры семейства. | Аналог FamilySymbol внутри .rfa. |
| **Идентификация типа** | `UniqueId` элемента. | Надёжен: FM сам создаёт .rvt, нет дублей из шаблонов. `ElementId` не используется — нестабилен при worksharing sync. |
| **Хранение .rvt** | `{family-id}/{version-label}/r{revit-version}/{sha256}.rvt` — аналогично .rfa. | Переиспользование существующей инфраструктуры storage. |
| **Версия Revit** | .rvt создаётся в текущей версии Revit. Одна версия = один .rvt файл. | Файлы .rvt не обратно совместимы. Хранение по версиям аналогично .rfa. |
| **ReadOnly** | Да, `FileAttributes.ReadOnly` на файле (ADR-016). | Аналогично .rfa файлам в managed storage. |

### Импорт

| Аспект | Решение | Обоснование |
|---|---|---|
| **Разбиение по категориям** | Автоматическое: FM группирует выделенные элементы по `BuiltInCategory`, создаёт отдельное семейство для каждой. | Удобство: выделил всё → получил несколько семейств. |
| **Имя семейства** | Пользователь задаёт в диалоге импорта (таблица с textbox). Дефолтное значение = имя категории. | Гибкость: можно создать несколько семейств труб с разными именами. |
| **Зависимости MEP** | Явный сбор: PipeType + Segment + PipeScheduleType + Material → один набор для `CopyElements`. | `CopyElements` не всегда автоматически копирует Segment и MEPSize. Явный обход графа зависимостей гарантирует полноту. |
| **Зависимости Host** | WallType/FloorType/RoofType → материалы слоёв копируются автоматически как зависимости CompoundStructure. | Дополнительный обход не нужен. |
| **Shared Parameters** | Не копируются, игнорируются. | Редкий кейс для системных типов. CopyElements не копирует Shared Parameters. |
| **Категория в FM** | Семейство загружается в "Без категории". Пользователь перемещает вручную. | Аналогично .rfa — пользователь контролирует организацию каталога. |

### Размещение

| Аспект | Решение | Обоснование |
|---|---|---|
| **Копирование в проект** | `ElementTransformUtils.CopyElements` из .rvt storage в активный проект. | Единственный программный способ переноса системных типов. |
| **Конфликт имён** | `IDuplicateTypeNamesHandler.UseDestinationTypes` — пропускаем, активируем существующий тип. | Не создаём дубликаты. |
| **Активация инструмента** | `uidoc.PostRequestForElementTypePlacement(elementType)`. | Revit автоматически выбирает нужный инструмент (Pipe, Wall, Duct). |
| **PipingSystemType** | Не контролируется FM. Пользователь выбирает систему вручную. | API не позволяет предустановить PipingSystemType для PostRequest. |
| **MEP размеры** | Не предустанавливаются FM. Пользователь выбирает размер через UI Revit. | Ограничение API Revit. |

### Атрибуты

| Аспект | Решение | Обоснование |
|---|---|---|
| **Извлечение** | Через существующую команду "Импорт данных" + пресеты категорий. | Переиспользование существующей инфраструктуры ADR-017. |
| **Механизм** | `OpenDocumentFile(.rvt)` → читать параметры `ElementType` напрямую. | Аналогично .rfa, но читаем ElementType вместо FamilyManager.Types. |
| **Отображение** | Вкладка "Атрибуты" (read-only), type selector — как у .rfa. | Единый UI для всех типов семейств. |

### Превью

| Аспект | Решение | Обоснование |
|---|---|---|
| **Превью** | Не генерируется автоматически. Пользователь добавляет вручную. | Системные типы не имеют 3D-превью как .rfa. Автогенерация сложна и не приоритетна. |

## Технические риски и митигации

### Риск 1: CopyElements не копирует все зависимости MEP

**Проблема**: `CopyElements` для PipeType может не скопировать Segment, PipeScheduleType и MEPSize автоматически, в отличие от ручного Ctrl+C/Ctrl+V.

**Митигация**: Явный обход графа зависимостей перед копированием:
```
PipeType
  → RoutingPreferenceManager.GetRules() → MEPSize → Segment
    → Segment.Material
    → Segment.ScheduleType (PipeScheduleType)
```
Все найденные ElementId добавляются в набор для `CopyElements`.

**Статус**: Требуется экспериментальная проверка на прототипе с трубами.

### Риск 2: Пустой проект без MEP-инфраструктуры

**Проблема**: `NewProjectDocument(UnitSystem.Metric)` создаёт проект без Segment, PipeScheduleType, MEP materials. Копирование PipeType в такой проект может создать тип с битыми ссылками.

**Митигация**: Варианты:
1. Если `CopyElements` копирует Segment как зависимость автоматически → пустой проект работает (ожидаемое поведение).
2. Если нет → fallback на `app.NewProjectDocument(app.DefaultProjectTemplate)` для MEP-типов.

**Статус**: Требуется экспериментальная проверка. Пользователь настаивает что CopyElements работает корректно с пустым проектом.

### Риск 3: DuplicateTypeNamesHandler и зависимости

**Проблема**: При `UseDestinationTypes` — тип пропускается, но зависимости (материалы) всё равно могут копироваться, создавая дубликаты.

**Митигация**: Использовать `IFailuresPreprocessor` для подавления предупреждений о дубликатах. Материалы с совпадающими именами будут пропущены автоматически.

**Статус**: Допустимый риск. Дубликаты материалов — некритичны (инвариант: материалы стандартизированы).

### Риск 4: Версионность .rvt

**Проблема**: .rvt файл версии R25 нельзя открыть в R24. Файл привязан к версии Revit.

**Митигация**: Хранение по версиям: `r{revit-major}/{sha256}.rvt`. При попытке загрузить тип в другой версии Revit → FM показывает сообщение "Нет совместимой версии файла для Revit {version}".

**Статус**: Решено. Аналогично существующей схеме для .rfa.

### Риск 5: OpenDocumentFile производительность

**Проблема**: Каждый "Загрузить и разместить" открывает .rvt файл (~200 КБ — 2 МБ) через `OpenDocumentFile`.

**Оценка**: 1-3 секунды на файл до 2 МБ. Приемлемо.

**Митигация**: При необходимости — кэширование открытого документа на время сессии (post-MVP).

**Статус**: Приемлемо для первой версии.

## Границы решения (Out of Scope)

- Автоматическое отслеживание изменений в исходных проектах.
- Генерация превью/иконок для системных семейств.
- Отображение MEP-размеров (сегментов) как под-узлов в дереве FM.
- Предустановка PipingSystemType / MechanicalSystemType перед размещением (ограничение API Revit).
- Предустановка диаметра/размера MEP перед размещением (ограничение API Revit).
- Копирование Shared Parameters между проектами.
- Удаление дублирующихся материалов при копировании между проектами.
- Удаление отдельных типоразмеров из системного семейства (только через UC-4 — полное обновление).
- FabricationPartType — не поддерживается `PostRequestForElementTypePlacement`.
- TransferProjectStandards через API — нет публичного API (запрос CF-882 с 2014 года).
- Кросс-версионное копирование (R25 → R24) — невозможно архитектурно.

## Схема БД: изменения

Для системных семейств минимально нужны изменения в существующих таблицах:

### `catalog_items` — дополнение

| Колонка | Тип | Описание |
|---|---|---|
| `family_source` | TEXT | `'loadable'` (по умолчанию) или `'system'`. Различает .rfa и .rvt семейства. |
| `revit_category` | TEXT | BuiltInCategory для системных семейств (например `'OST_PipeCurves'`). NULL для .rfa. |

### `family_types` — без изменений

Существующая схема хранит `type_name` + `catalog_item_id`. UniqueId типа хранится отдельно — добавляется колонка:

| Колонка | Тип | Описание |
|---|---|---|
| `type_unique_id` | TEXT | UniqueId ElementType внутри .rvt файла. NULL для .rfa. |

### `family_files` — без изменений

Расширение файла определяется по `file_name` (.rfa / .rvt).

## Критерии приёмки

1. Пользователь может выделить элементы в любом открытом проекте Revit и импортировать их системные типы как семейства в FM.
2. FM автоматически разбивает выделенные элементы по категориям и показывает диалог с таблицей для задания имён.
3. FM создаёт чистый .rvt файл минимального размера для каждого семейства.
4. Имя семейства в FM соответствует введённому пользователем в диалоге импорта.
5. Импортированное семейство отображается в каталоге FM с типоразмерами внутри, в категории "Без категории".
6. Пользователь может загрузить и разместить конкретный типоразмер в активном проекте через стандартный инструмент Revit.
7. Системный тип корректно копируется в проект вместе с зависимостями (материалы, Segment, фиттинги).
8. Если тип уже существует в проекте — FM активирует существующий, не создавая дубликатов.
9. Атрибуты типоразмера извлекаются командой "Импорт данных" через пресеты категорий и отображаются в окне свойств.
10. Повторный импорт с именем существующего семейства создаёт новую версию с дополнительными/обновлёнными типоразмерами.
11. Пользователь может загрузить тип без размещения (Load Only) — тип доступен через Type Selector.
12. Поддерживаются все основные системные семейства: стены, перекрытия, трубы, воздуховоды, лотки, короба, крыши.
13. Файлы .rvt хранятся в managed storage с ReadOnly-флагом, аналогично .rfa.

## Эксперименты для прототипа

Перед реализацией MVP необходимо экспериментально подтвердить:

| # | Гипотеза | Метод проверки |
|---|---|---|
| E-1 | `CopyElements` корректно копирует PipeType + все зависимости (Segment, MEPSize, PipeScheduleType, фиттинги) в пустой проект `NewProjectDocument(UnitSystem.Metric)`. | Создать тест: выделить трубу → NewProjectDocument → CopyElements → проверить RoutingPreferenceManager. |
| E-2 | `CopyElements` для PipeType из чистого .rvt в реальный проект корректно переносит все MEP-зависимости. | Создать чистый .rvt с PipeType → OpenDocumentFile → CopyElements в реальный проект → проверить тип. |
| E-3 | `PostRequestForElementTypePlacement` корректно активирует Pipe Tool / Duct Tool / Conduit Tool / CableTray Tool. | Вызвать для каждого MEP-типа, убедиться что Revit показывает нужный инструмент. |
| E-4 | UniqueId стабилен при сохранении/переоткрытии чистого .rvt. | Создать .rvt → записать UniqueId → закрыть → открыть → проверить. |
| E-5 | `CopyElements` для WallType корректно копирует CompoundStructure с материалами слоёв. | Копировать WallType → проверить слои и материалы в целевом проекте. |
