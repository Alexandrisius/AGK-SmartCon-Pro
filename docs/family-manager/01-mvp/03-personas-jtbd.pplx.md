# FamilyManager Personas and Jobs To Be Done

## Цель документа

Документ описывает пользователей FamilyManager и реальные задачи, ради которых строится MVP. Это нужно, чтобы не проектировать функции ради функций.

## Primary Persona: Инженер / проектировщик

| Поле | Описание |
| --- | --- |
| Контекст | Работает в Revit, часто ищет семейства для текущего проекта |
| Боль | Семейства лежат в папках, названия не всегда понятные, актуальность неизвестна |
| MVP-ценность | Быстрый поиск и загрузка семейства без ручного прохода по папкам |
| Критичный сценарий | Найти семейство → посмотреть карточку → загрузить в проект |
| Не интересует в MVP | Governance, approval, server admin |

## Primary Persona: BIM-координатор

| Поле | Описание |
| --- | --- |
| Контекст | Поддерживает библиотеку команды или отдела |
| Боль | Невозможно понять, какие семейства проверены, какие устарели, какие дублируются |
| MVP-ценность | Импорт, теги, статусы, базовая структура каталога |
| Критичный сценарий | Импорт папки → классификация → статус Verified/Draft → передача локального каталога |
| Не интересует в MVP | Полноценный RBAC и approval chain |

## Secondary Persona: BIM-менеджер

| Поле | Описание |
| --- | --- |
| Контекст | Отвечает за стандарты BIM-контента в компании |
| Боль | Нужна будущая корпоративная платформа, но нельзя начинать с огромного сервера |
| MVP-ценность | Проверить доменную модель, статусы, версионирование, provider abstraction |
| Критичный сценарий | Убедиться, что local catalog потом масштабируется в corporate catalog |
| Не интересует в MVP | Ручная работа с каждым файлом, если нет стратегии роста |

## Technical Persona: Разработчик smartCon

| Поле | Описание |
| --- | --- |
| Контекст | Развивает smartCon как multi-version Revit plugin |
| Боль | Новые модули могут нарушить dependency rule, Revit threading, net48 compatibility |
| MVP-ценность | Чёткие контракты, тестируемые Core-модели, изоляция Revit API |
| Критичный сценарий | Добавить модуль без регрессий в R19/R21/R24/R25/R26 |
| Не интересует в MVP | Сложные framework-миграции и лишние зависимости |

## Jobs Map

| Job | Пользователь | MVP Priority | Outcome |
| --- | --- | --- | --- |
| Импортировать семейство | Инженер, BIM-координатор | P0 | Запись появилась в каталоге |
| Импортировать папку | BIM-координатор | P0 | Создан batch-report |
| Найти семейство | Инженер | P0 | Нужная запись найдена за несколько действий |
| Проверить пригодность | Инженер | P0 | Видны категория, статус, версия, путь, типы |
| Загрузить в проект | Инженер | P0 | Семейство доступно в Revit |
| Пометить статус | BIM-координатор | P1 | Draft/Verified/Deprecated/Archived отражены в каталоге |
| Обнаружить дубликаты | BIM-координатор | P1 | Hash показывает повторяющиеся файлы |
| Подготовить к серверу | BIM-менеджер | P1 | Provider abstraction не привязана к SQLite |
| Проверить legacy build | Разработчик | P0 | R19/R21/R24 не падают при сборке |

## Anti-Personas for MVP

| Пользователь | Почему не MVP |
| --- | --- |
| Корпоративный администратор SSO | Требует server phase |
| Marketplace moderator | Требует public sharing и legal model |
| Data analyst | Нужна usage telemetry и сервер |
| AI-search пользователь | Нужен корпус данных и оценка качества поиска |
