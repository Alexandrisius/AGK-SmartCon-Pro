## Проблема 1: Подбор размера только по целевому коннектору (тройники, крестовины)

### Описание проблемы

**Текущее поведение:** При S4-подборе алгоритм строит `LookupColumnConstraint` от **других** коннекторов
и фильтрует строки CSV, требуя что **другие** коннекторы остались неизменными. Только целевой коннектор
может изменить свой DN.

**Реальный кейс:** Тройник DN 65×DN 65 (проход) × DN 65 (ответвление), труба DN 50.

| Целевой коннектор | Что делает текущий алгоритм | Ожидание пользователя |
|---|---|---|
| Ответвление (conn3) | Находит строку DN 65×DN 50 — **успешно**, меняет на DN 50×DN 65×DN 50 | DN 50×DN 50×DN 50 (или ближайший с DN 50 на ответвлении) |
| Проходной (conn1/conn2) | Ищет строку где conn1=DN 50 **и conn2=DN 65** — **нет такой строки!** Предлагает reducer | DN 50×DN 50×DN 50 (или ближайший с DN 50 на проходе) |

**Суть:** Пользователь, выбравший конкретный коннектор для присоединения, **согласен** с тем, что
все остальные коннекторы динамического семейства тоже изменятся. Семейство — единое целое.

### Бизнес-кейсы

#### Кейс 1A: Тройник DN 65×65, присоединение ответвлением к трубе DN 50

```
Таблица поиска тройника:
  DN_Проход | DN_Ответвление
  65        | 65
  65        | 50
  50        | 50
  50        | 32
  40        | 40
  32        | 32

Текущий behaviour:
  constraints = [DN_Проход = 65]  ← от conn1 (проход)
  → строки: только DN_Проход=65 → DN_Ответвление ∈ {65, 50}
  → target DN 50 найден в строке 65×50 → УСПЕХ

Ожидаемый behaviour (тот же результат):
  Без constraints по другим коннекторам
  → Найти строку где DN_Ответвление ближе всего к DN 50 → 65×50 или 50×50
  → Выбрать 50×50 (целевой=50, суммарное отклонение минимально)
```

#### Кейс 1B: Тройник DN 65×65, присоединение ПРОХОДНЫМ коннектором к трубе DN 50

```
Текущий behaviour:
  constraints = [DN_Ответвление = 65]  ← от conn3
  → строки: только DN_Ответвление=65 → DN_Проход ∈ {65}
  → target DN 50 НЕ НАЙДЕН с constraint → "нужен reducer"
  → НЕВЕРНО! Строка 50×50 существует!

Ожидаемый behaviour:
  Без constraints → ищем ближайшую строку где DN_Проход ближе всего к DN 50
  → строка 50×50: DN_Проход=50, DN_Ответвление=50 → точное совпадение!
  → Применить: тройник станет DN 50×50×50
```

#### Кейс 1C: Крестовина DN 65×65×65×65, присоединение к трубе DN 50

```
Текущий behaviour:
  constraints от 3 других коннекторов — все 65
  → единственная строка: 65×65×65 → target 50 не найден → reducer

Ожидаемый behaviour:
  Без constraints → ищем строку где целевой коннектор ≈ DN 50
  → если есть 50×50×50×50 → точное совпадение, все коннекторы меняются
```

### Решение

#### Этап 1: Глобальный поиск по типоразмерам (без фиксации других коннекторов)

**Изменить `BuildResolutionPlan()` в PipeConnectCommand.cs:**

1. **Вариант A (рекомендуемый):** Добавить двухпроходную стратегию:
   - **Проход 1 (текущий):** Поиск с constraints от других коннекторов — если найден точный, используем его
   - **Проход 2 (новый):** Если точного нет — поиск БЕЗ constraints, минимизируя суммарное отклонение всех коннекторов от target

2. **Изменить `BuildMultiColumnConstraints()`:** Не просто строить constraints, а формировать
   стратегию поиска: «сначала с фиксацией, потом без».

3. **Добавить новый метод `FindBestSizeRow()` в `ILookupTableService`/`RevitLookupTableService`:**
   ```
   Вход: CSV таблица, целевой столбец, target DN
   Выход: строка CSV где целевой столбец максимально близок к target
   ```

4. **Результат поиска БЕЗ constraints включает новые радиусы ВСЕХ коннекторов.**
   Это нужно для корректного обновления downstream-цепочки (Chain+).

#### Этап 2: Новая модель результата подбора

Расширить `ParameterResolutionPlan`:
```csharp
private sealed record ParameterResolutionPlan(
    bool   Skip,
    double TargetRadius,
    bool   ExpectNeedsAdapter,
    string? WarningMessage,
    IReadOnlyList<LookupColumnConstraint> LookupConstraints,
    // NEW:
    bool   ConstrainedOtherConnectors,  // true = другие коннекторы тоже изменятся
    Dictionary<int, double>? AllConnectorTargetRadii  // целевой радиус каждого коннектора
);
```

#### Этап 3: Обновление Init() / ChangeDynamicSize()

При применении размера, который меняет другие коннекторы:
1. Установить целевой радиус для выбранного коннектора
2. Перечитать ВСЕ коннекторы после Regenerate
3. Если у downstream-элементов цепочки целевые коннекторы изменились — пересоединить их

## Проблема 2: Формат выпадающего списка — показать полную конфигурацию типоразмера

### Описание проблемы

**Текущее поведение:** Dropdown показывает просто `DN 25`, `DN 32`, `DN 40` — размер только
**выбранного** коннектора. Для многопортовых семейств непонятно, как изменятся **остальные** коннекторы.

**Ожидание:** Пользователь видит полную конфигурацию: `DN 65 × DN 50 × DN 65` (первым — целевой коннектор),
и при выборе вся конфигурация применяется к семейству целиком.

### Бизнес-кейсы

#### Кейс 2A: Тройник — пользователь видит все доступные конфигурации

```
Выпадающий список:
  АВТОПОДБОР (DN 65 × DN 65 × DN 65)
  DN 65 × DN 65 × DN 50     ← ответвление уменьшено
  DN 65 × DN 50 × DN 65     ← один проход уменьшен
  DN 50 × DN 50 × DN 50     ← все уменьшены
  DN 50 × DN 50 × DN 32
  DN 40 × DN 40 × DN 40
  DN 32 × DN 32 × DN 32

Формат: ЦЕЛЕВОЙ × ДРУГИЕ (в порядке ConnectorIndex)
```

#### Кейс 2B: Кран шаровый переходной (2 коннектора, разные DN)

```
Выпадающий список:
  АВТОПОДБОР (DN 20 × DN 25)
  DN 20 × DN 25
  DN 20 × DN 20
  DN 25 × DN 32
  DN 15 × DN 20

Первое число = целевой коннектор (который присоединяется к static)
Второе число = другой коннектор
```

#### Кейс 2C: Обычный кран (2 коннектора, один DN)

```
Выпадающий список:
  АВТОПОДБОР (DN 50)
  DN 50
  DN 40
  DN 32
  DN 25

Оба коннектора всегда DN = N — показываем просто DN
```

### Решение

#### Этап 1: Новая модель `SizeOption` → `FamilySizeOption`

```csharp
public sealed record FamilySizeOption
{
    /// <summary>Отображаемое имя: "DN 65 × DN 50 × DN 65"</summary>
    public required string DisplayName { get; init; }

    /// <summary>Радиус ЦЕЛЕВОГО коннектора (для обратной совместимости).</summary>
    public required double Radius { get; init; }

    /// <summary>Радиусы ВСЕХ коннекторов по индексам.</summary>
    public IReadOnlyDictionary<int, double> AllConnectorRadii { get; init; } = new Dictionary<int, double>();

    /// <summary>Источник: "LookupTable", "FamilySymbol", "PipeType", ""</summary>
    public string Source { get; init; } = "FamilySymbol";

    /// <summary>Флаг "АВТОПОДБОР"</summary>
    public bool IsAutoSelect { get; init; }

    /// <summary>Имя типоразмера (FamilySymbol.Name) если применимо.</summary>
    public string? SymbolName { get; init; }
}
```

**Правила формирования DisplayName:**
1. Если все коннекторы имеют одинаковый DN → `"DN {n}"`
2. Если коннекторы имеют разные DN → `"DN {target} × DN {other1} [× DN {other2} ...]"`
   - Целевой коннектор ВСЕГДА первый
   - Остальные в порядке ConnectorIndex

#### Этап 2: Новый метод `GetAvailableFamilySizes()` в `IDynamicSizeResolver`

```csharp
/// <summary>
/// Возвращает все доступные типоразмеры FamilyInstance с радиусами ВСЕХ коннекторов.
/// Первый элемент = текущая конфигурация (АВТОПОДБОР).
/// </summary>
IReadOnlyList<FamilySizeOption> GetAvailableFamilySizes(
    Document doc, ElementId elementId, int targetConnectorIndex,
    IReadOnlyList<LookupColumnConstraint>? constraints = null);
```

**Алгоритм (RevitDynamicSizeResolver):**

**Для LookupTable-семейств:**
1. Прочитать CSV
2. Для каждой уникальной строки извлечь значения ВСЕХ query-столбцов
3. Каждая строка → `FamilySizeOption` с `AllConnectorRadii` для каждого коннектора
4. Отсортировать: сначала ближайшие к текущему целевому коннектору, потом по DN

**Для FamilySymbol-семейств:**
1. Для каждого FamilySymbol: SubTransaction → ChangeTypeId → Regenerate
2. Прочитать радиусы ВСЕХ коннекторов
3. Каждая конфигурация → `FamilySizeOption`

**Для Pipe/MEPCurve:**
1. Без изменений — только один коннектор, `FamilySizeOption` эквивалентен `SizeOption`

#### Этап 3: Обновление `LoadDynamicSizes()` в ViewModel

1. Вызывать `GetAvailableFamilySizes()` вместо `GetAvailableSizes()`
2. Формировать `DisplayName` по правилам выше
3. Дедупликация по полному `AllConnectorRadii` (не по DisplayName)

#### Этап 4: Обновление `ChangeDynamicSize()` в ViewModel

При выборе `FamilySizeOption` с несколькими коннекторами:
1. Применить радиус целевого коннектора через `TrySetConnectorRadius`
2. После Regenerate — проверить ВСЕ коннекторы
3. Если радиусы других коннекторов не соответствуют ожидаемым — это нормально (семейство
   применило конфигурацию целиком через lookup-таблицу)
4. Обновить downstream-элементы цепочки если их parent-коннекторы изменились

## Проблема 3: Обратная связь при смене типоразмера Revit + учёт управляющих параметров

### Описание проблемы

**Ситуация:** Семейство может иметь Revit-типоразмеры (FamilySymbol) с разными размерами коннекторов.
Например, «Кран шаровой» с типами:
- `ТИП_1 — С ручкой` — доступные DN: 15, 20, 25, 32, 40, 50, 65
- `ТИП_2 — С электроприводом` — доступные DN: 25, 32, 40

**Текущее поведение:** При переборе FamilySymbol в `TryChangeTypeTo()` или `GetFamilySymbolSizes()`
система перебирает **все** типы и показывает **все** доступные DN, не разделяя по управляющим параметрам.

**Проблемы:**
1. Пользователь видит DN 20 в списке, но при выборе — тип меняется с «С электроприводом» на «С ручкой»,
   а пользователь этого не понимает.
2. Lookup-таблица может иметь вводные параметры, не связанные с размером коннектора
   (например `ТИП_УПРАВЛЯЮЩЕГО_ОРГАНА`), которые ограничивают доступные DN.

### Бизнес-кейсы

#### Кейс 3A: Кран шаровой с ручным/электроприводным управлением

```
Семейство: «Кран шаровой»
Lookup-таблица:
  ТИП_ОРГАНА | DN
  1 (ручка)  | 15
  1 (ручка)  | 20
  1 (ручка)  | 25
  1 (ручка)  | 32
  1 (ручка)  | 40
  1 (ручка)  | 50
  2 (привод) | 25
  2 (привод) | 32
  2 (привод) | 40

Текущий элемент: ТИП_2 (электропривод), DN 25
Цель: Присоединить к трубе DN 40

Ожидаемый список:
  АВТОПОДБОР (DN 25, Электропривод)
  DN 40 (Электропривод)         ← остались в текущем типе органа
  DN 32 (Электропривод)
  DN 25 (Электропривод)
  DN 50 (Ручка)                 ← смена типа органа!
  DN 40 (Ручка)
  DN 32 (Ручка)
  DN 25 (Ручка)
  DN 20 (Ручка)
  DN 15 (Ручка)

При выборе DN 50 (Ручка) → инфо: "Тип управляющего органа изменён: Электропривод → Ручка"
```

#### Кейс 3B: Семейство с Revit-типоразмерами (без LookupTable)

```
Семейство: «Клапан обратный»
Типы Revit:
  "M016_WaterMeter_WaterG5_20mm"     → DN 20
  "M016_WaterMeter_WaterG5_25mm"     → DN 25
  "M016_WaterMeter_WaterG5_Counter20" → DN 20 (счётчик)
  "M016_WaterMeter_WaterG5_Counter25" → DN 25 (счётчик)

Текущий: "M016..._20mm" (без счётчика)
Цель: DN 25

Ожидаемый список:
  АВТОПОДБОР (DN 20, Без счётчика)
  DN 25, Без счётчика         ← "M016...25mm"
  DN 20, С счётчиком          ← смена подтипа!
  DN 25, С счётчиком          ← смена подтипа!

Инфо при выборе "DN 25, С счётчиком":
  "Типоразмер изменён: M016_WaterMeter_WaterG5_20mm → M016_WaterMeter_WaterG5_Counter25"
```

### Решение

#### Этап 1: Определение «управляющих» параметров (non-size query parameters)

**В `FamilyParameterAnalyzer` или новом методе:**

1. После парсинга `size_lookup(TableName, target, default, Q1, Q2, ...)`:
   - `Q1` может быть DN (связан с размером коннектора)
   - `Q2` может быть `ТИП_ОРГАНА` (не связан с размером)
2. Определить: какие query-параметры связаны с `CONNECTOR_RADIUS`/`CONNECTOR_DIAMETER`?
   - Тот, для которого через `FamilyParameterAnalyzer` найдена цепочка → размерный
   - Остальные → управляющие (non-size)
3. Запомнить в контексте: `NonSizeQueryParameters: List<string>`

#### Этап 2: Группировка доступных размеров по управляющим параметрам

**В `RevitDynamicSizeResolver`:**

1. При извлечении строк CSV — кроме значений целевого столбца, извлечь также
   значения управляющих столбцов
2. Группировать по значениям управляющих параметров
3. Текущая группа (значения управляющих параметров, совпадающие с текущим элементом) — первая

**Для FamilySymbol-семейств:**
1. Определить текущее значение управляющих параметров элемента
2. Группировать FamilySymbol по значениям этих параметров
3. Текущая группа — первая

#### Этап 3: Новая модель `FamilySizeOption` — поля для обратной связи

```csharp
public sealed record FamilySizeOption
{
    // ... (поля из этапа 2 проблемs 2)

    /// <summary>
    /// Имя типоразмера FamilySymbol если applicable.
    /// null для LookupTable-семейств.
    /// </summary>
    public string? SymbolName { get; init; }

    /// <summary>
    /// Текущий типоразмер (до смены). Если не совпадает с SymbolName — будет смена типа.
    /// </summary>
    public string? CurrentSymbolName { get; init; }

    /// <summary>
    /// Значения управляющих (non-size) параметров для данной конфигурации.
    /// Ключ = имя параметра, Значение = значение из CSV или FamilySymbol.
    /// </summary>
    public IReadOnlyDictionary<string, string> NonSizeParameterValues { get; init; }
        = new Dictionary<string, string>();

    /// <summary>
    /// true если при выборе данной опции управляющие параметры изменятся
    /// (текущие значения не совпадают с целевыми).
    /// </summary>
    public bool RequiresNonSizeParameterChange { get; init; }
}
```

#### Этап 4: Обратная связь в UI — информационное поле

**Обновить `PipeConnectEditorViewModel`:**

1. При выборе размера из dropdown — вычислить diff между текущим и целевым:
   - Смена типоразмера: `"Типоразмер: {current} → {target}"`
   - Смена управляющего параметра: `"Тип органа: Электропривод → Ручка"`
   - Смена нескольких коннекторов: `"Размеры коннекторов: DN 65×65×65 → DN 50×50×50"`

2. Отображать в `StatusMessage` или в новом информационном поле.

**Обновить `PipeConnectEditorView.xaml`:**

Добавить TextBlock или ToolTip к ComboBox размеров:
```xml
<!-- Информация о смене типа -->
<TextBlock Grid.Row="X"
           Text="{Binding SizeChangeInfo}"
           FontSize="10" Foreground="#E65100"
           TextWrapping="Wrap"
           Visibility="{Binding HasSizeChangeWarning, Converter={StaticResource BoolToVisibility}}"/>
```

#### Этап 5: DisplayName с указанием управляющих параметров

**Правила:**
1. Если `NonSizeParameterValues` совпадают с текущими → суффикс не нужен
2. Если отличаются → добавить суффикс: `DN 50 (Ручка)` вместо просто `DN 50`
3. При нескольких несовпадающих параметрах — перечислить через запятую:
   `DN 50 (Ручка, Серия 2)`

1. Создать `FamilySizeOption` модель
2. Создать `GetAvailableFamilySizes()` в `RevitDynamicSizeResolver`
3. Обновить `LoadDynamicSizes()` и `ChangeDynamicSize()` в ViewModel
4. Формат DisplayName: `"DN {target} × DN {other1} × ..."`
5. Тесты: тройник, кран переходной, крестовина

### Фаза 3: Обратная связь по управляющим параметрам (Проблема 3)
**Приоритет: Средний**

1. Определить non-size query parameters в `FamilyParameterAnalyzer`
2. Группировка размеров по управляющим параметрам
3. DisplayName с суффиксом типа: `"DN 50 (Ручка)"`
4. Информационное поле в UI — предупреждения о смене типа
5. Тесты: кран с ручкой/электроприводом, клапан с типами

## Детальные бизнес-кейсы использования нового функционала

### БК-1: Тройник к трубе — умный подбор

**Сценарий:** В модели тройник DN 65×65×65. Пользователь хочет присоединить его к трубе DN 50.

**Шаги:**
1. Пользователь кликает тройник (dynamic) — выбирает проходной коннектор (conn1, DN 65)
2. Пользователь кликает трубу DN 50 (static)
3. **Старый:** Плагин находит что DN 65→50 невозможно с фиксированным ответвлением → предлагает reducer
4. **Новый:** Плагин находит типоразмер DN 50×50×50 в таблице → применяет напрямую
5. Dropdown показывает: `АВТОПОДБОР (DN 50 × DN 50 × DN 50)`
6. Пользователь доволен — нажимает «Соединить» — прямое соединение без reducer

### БК-2: Тройник к трубе — выбор из dropdown

**Сценарий:** Тройник DN 65×65×65. Пользователь хочет вручную подобрать размер.

**Шаги:**
1. После двух кликов открывается окно редактора
2. Dropdown показывает:
   ```
   АВТОПОДБОР (DN 65 × DN 65 × DN 65)
   DN 65 × DN 65 × DN 50
   DN 65 × DN 50 × DN 65
   DN 50 × DN 50 × DN 50
   DN 50 × DN 50 × DN 32
   DN 40 × DN 40 × DN 40
   DN 32 × DN 32 × DN 32
   ```
3. Пользователь выбирает `DN 50 × DN 50 × DN 32`
4. Тройник меняется на DN 50×50×32
5. Info: `"Размеры: DN 65×65×65 → DN 50×50×32"`

### БК-3: Кран шаровый переходной — корректный показ

**Сценарий:** Кран переходной DN 20×25. Пользователь присоединяет DN 20-коннектором к трубе DN 25.

**Шаги:**
1. Dropdown показывает:
   ```
   АВТОПОДБОР (DN 25 × DN 20)     ← целевой коннектор первый
   DN 25 × DN 25
   DN 25 × DN 20
   DN 20 × DN 25
   DN 20 × DN 20
   DN 32 × DN 25
   DN 32 × DN 20
   ```
2. Пользователь выбирает `DN 25 × DN 25`
3. Оба коннектора крана становятся DN 25

### БК-4: Кран с ручкой/электроприводом — учёт управляющего параметра

**Сценарий:** Кран DN 25 (электропривод). Присоединяется к трубе DN 50.

**Шаги:**
1. Таблица поиска содержит: `ТИП_ОРГАНА | DN`
2. Текущее значение `ТИП_ОРГАНА = 2` (электропривод)
3. Для электропривода DN 50 нет! Но для ручки (ТИП_ОРГАНА=1) — есть.
4. Dropdown показывает:
   ```
   АВТОПОДБОР (DN 25) [Электропривод]
   DN 40 [Электропривод]
   DN 32 [Электропривод]
   DN 25 [Электропривод]
   ─────────────────────
   DN 50 [Ручка]          ← визуально отделены
   DN 40 [Ручка]
   DN 32 [Ручка]
   DN 25 [Ручка]
   DN 20 [Ручка]
   DN 15 [Ручка]
   ```
5. При наведении на `DN 50 [Ручка]`: tooltip `"Тип органа изменится: Электропривод → Ручка"`
6. При выборе: StatusMessage = `"Тип управляющего органа: Электропривод → Ручка. Размер: DN 25 → DN 50"`

### БК-5: Семейство по Revit-типам — информирование о смене типа

**Сценарий:** Клапан обратный типа `"ТИП_1"` DN 20. Присоединяется к трубе DN 25.

**Шаги:**
1. Плагин перебирает все FamilySymbol
2. Находит `"ТИП_2"` DN 25 — подходит
3. Dropdown показывает:
   ```
   АВТОПОДБОР (DN 20) [ТИП_1]
   DN 25 [ТИП_2]          ← тип изменится
   DN 20 [ТИП_2]          ← тип изменится
   DN 25 [ТИП_1]          ← текущий тип
   DN 20 [ТИП_1]          ← текущий тип
   ```
4. StatusMessage: `"Типоразмер: ТИП_1 → ТИП_2. Размер: DN 20 → DN 25"`

### БК-6: Цепочка с тройником — корректное обновление downstream

**Сценарий:** Dynamic = тройник DN 65×65×65, цепочка = труба DN 65 → отвод DN 65.
Присоединяется к трубе DN 50 через проходной коннектор.

**Шаги:**
1. S4 подбирает DN 50×50×50 (без фиксации ответвления)
2. Init() применяет: тройник → DN 50×50×50
3. Chain+ уровень 1 (труба): parent=тройник conn2=DN 50, child=труба DN 65
   - TrySetConnectorRadius(труба, DN 50) — труба меняет диаметр на DN 50
4. Chain+ уровень 2 (отвод): parent=труба DN 50, child=отвод DN 65
   - TrySetConnectorRadius(отвод, DN 50) — отвод меняется на DN 50
5. Все соединяется без reducer-ов

## Риски и ограничения

1. **LookupTable без связи «параметр ↔ коннектор»:** Не для всех query-параметров можно
   однозначно определить какой коннектор они контролируют. Маппинг столбец→коннектор
   основан на имени параметра и его цепочке зависимостей. Если имена нестандартные —
   может потребоваться эвристика по значению.

2. **Производительность:** Полный перебор строк CSV для многопортовых семейств с десятками
   типоразмеров может быть медленным. Оптимизация: кешировать результаты на уровне сессии.

3. **Non-size параметры определены не всегда:** Если формула параметра не парсится через
   FormulaSolver (спецсимволы, нестандартный синтаксис) — non-size параметры не будут
   определены. Fallback: показать все DN без группировки.

4. **Семейства с 3+ query-параметрами:** Крестовина с 4 коннекторами может иметь
   `size_lookup(T, val, def, DN1, DN2, DN3, DN4)`. Dropdown будет показывать
   `DN {n1} × DN {n2} × DN {n3} × DN {n4}` — может быть длинным. Ограничение: обрезать
   до разумного максимума или использовать многострочный формат.
